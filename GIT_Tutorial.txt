>Ao primeiro acesso, cadastrar e-mail e usuario global:
$ git config --global user.mail "alexandrechecoli@gmail.com"
$ git config --global user.name "Alexandre Checoli"

>Passar os arquivos para  "staging a	rea" (é um intermediário)
$ git add nome_do_arquivo

Adicionar todos os arquivos que tenham .txt
$ git add *.txt

Adicionar TODOS os arquivos da pasta
$ git add .

Para slavar as mudanças no projeto "comitar"
$ git commit -m "Nome do commit"

>Inserir arquivos que o git năo deve trackear
Criar um arquivo com o nome ".gitignore", e dentro dele, todos os arquivos que devem ser
ignorados do "tracking" -> Sempre UMA INFORMAÇĂO POR LINHA

>É necessário inicializar um repositório na pasta.
$ git init

Para se verificar o que foi alterado em um arquivo (quando o status aponto alteraçőes)
$ git diff

Para verificar o que foi alterado nos arquivos que estăo n "staging area" (tudo que foi alterado
e ainda năo foi comitado)
$ git diff --staged 

Visualizar todos os commits já feitos. ( a chave que aparece ao lado do commit é a valor usado quando 
se deseja reaver um determinado commit)
$git log

Visualizar os commits só pelos nomes
$ git log prettyone==oneline

Mostrar a GUI do git
$ gitk

Quando se faz um commit antes da hora e se deseja "alterar o commit" ou seja, editar um commit.
$ git commit --amend -m "Ediçăo de commit já existente"

Realizar testes em arquivos (sem usar o add, com os arquivos modificados mas nao adicionados)
, e depois voltar para a versăo do último commit
$ git checkout -- README.txt


Quando arquivos forem removidos, para que o commit funcione, é necessário removę-los também da stage area
$ git rm "Nome do arquivo"
OBS: Nesta versăo, só com o add ele já removeu o que tinha de ser removido.

Criar TAGS, por exemplo, uma TAG pode ser uma versăo v0.1 (sempre para o commit atual)
$ git tag -a v1.0 -m "Versăo 1.0"

Criar uma TAG para um commit antigo
$ git tag -a v0.0 chave_do_commit -m "Comentario da tag"

Depois de criar as tags,é possível recuperar todos os arquivos desta tag
$ git ckeckout nome_da_tag
$ git ckeckout v1.0
Com isso, os arquivos serăo todos retomados para a versăo anterior

Para voltar a versăo atual.
$ git checkout master

Para deletar uma TAG
$ git tag -d nome_da_tag
$ git tag -d v0.0

Criar BRANCHES. Com um branch criado, vários commits podem ser feitos a partir de um ponto, em que năo serăo afetados
por commits de outras tags. Permite criar várias ramificaçőes diferentes. Por padrăo, o git sempre cria um branch padrăo que é 
o MASTER
$ git branch nome_do_branch
$ git branch teste

Para começar a trabalhar em algum branch
$ git checkout nome_do_branch

OBS: Sempre comitar as alteraçőes feitas em um branch, caso contrário quando se passar de um para outro o ambiente
continua o mesmo.
Quando for necessário "unir" os branches, ou seja, supondo que o branch teste foi criado, e por um tempo trabalhado em 
paralelo ao master para ser validado. Ao fim da validaçao, o branch testes pode ser juntado ao master novamente. isso é
feito com o comando merge, OBS: vc deve ESTAR no branch que RECEBERÁ as alteraçőes, por exemplo, estando no branch master, 
o comando.
$ git merge teste 

irá trazer as informaçőes de TESTE para o MASTER.

Remover um branch
$ git branch -d nome_do_branch
$ git branch -d testes

Para "Sincronizar" um repositorio local com um já existente do GitHub (ao escolher caminho HTTPS colocar
aspas "" no nome:
$ git remote add origin https://github.com/angelicamariatortola/academic.git

Mandar os arquivos para o "servidor" (passe tudo do master(atual) para o origin(internet))
$ git push origin master

Verificar se um repositório está "emparelhado" com um repo online, ou somente local
$ git remote -v

Para remover um repositório remoto e ficar só com o local novamente
$ git remote remove origin

Para remover o git da pasta.
Basta remover a pasta 'oculta' do git. No windows, entrar no cmd no local do git, e digitar
$ rmdir .git (se for uma pasta só)
$ rmdir /s .git (se tiver subpastas)

Para realizar o "push" de um novo branch criado:
$ git push -u origin nome_do_branch

 ################# Para trabalhar em equipe: #####################

1 - O gerenciador (A) do projeto cria um novo projeto.

2 - Quem deseja contribuir (B) com ele deve criar um "fork" (ir no projeto e clicar em "fork")
    Isso vai criar uma "cópia" do projeto nos repositórios de (B), sempre que A fizer algum commit, B vai poder vizualizar (somente no site)

3 - B deve clonar o repositorio "Forkado" para a área de trabalho.

4 - Para B realizar commits em A, deve se usar o push normalmente (push origin master), isso levará os commits para o fork, com isso, ir no site e criar um "pull request", um pedido para que A aceite os commits.

5 - A deve visualizar o pedido e aceitar ou rejeitar.

6 - Para que B mantenha o fork atualizado, deve-se colocar o "upstream" como o diretório de A
$git remote add upstream URL_diretório_de_A
Em seguida, o pull deve ser realizado do upstream
$git pull upstream master
